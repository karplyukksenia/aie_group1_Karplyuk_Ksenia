# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- Использовалась база знаний по основам Python (25 коротких документов).
- Чанкинг: по словам, `chunk_size=25`, `overlap=5`.
- Эмбеддинги: `sentence-transformers/all-MiniLM-L6-v2` (размерность 384).
- Построен FAISS-индекс `IndexFlatIP`.
- Retrieval показал отличное качество: `Hit@3 = 1.0`, `Hit@5 = 1.0`, `Recall@5 = 1.0` на 10 контрольных запросах.
- Добавлено 3 новых документа про `set` → после переиндексации retrieval корректно начал возвращать новые документы.
- Реализован простой mini-RAG (retrieval → контекст → ответ + источники).

## 2. Среда и воспроизводимость

- Python: 3.12
- faiss: faiss-gpu-cu12
- sentence-transformers: последняя версия
- torch: 2.x (GPU)
- Устройство: GPU (T4)
- Seed: 17 (зафиксирован для `random`, `numpy`, `torch`)
- Как запустить: открыть `HW14.ipynb` и выполнить Run All.

Документы тематически связаны, но имеют чёткое смысловое разделение, что делает их идеальными для задачи retrieval и mini-RAG. ## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Embedding-модель: `sentence-transformers/all-MiniLM-L6-v2`
- Метрика сходства: косинусное (векторы нормализованы + `IndexFlatIP`)
- Индекс FAISS: `IndexFlatIP`
- `top_k` по умолчанию: 3 и 5 (для оценки) / 3 (для mini-RAG)
- Первые примеры retrieval показали высокую релевантность.

### 4.2. Контрольные запросы и оценка retrieval

- Количество контрольных запросов: 10
- Релевантным считался документ с `expected_doc_id`
- Метрики: `hit@k`, `recall@k`
- Итоговый `Hit@3`: **1.000** (10/10)
- Итоговый `Hit@5`: **1.000** (10/10)
- `Recall@5`: **1.000**
- Самые простые запросы: про lists, tuples, functions.
- Проблемных запросов в базовой версии не было.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- Сравнивался параметр `chunk_size` (25 vs 5 слов).
- Вариант A (основной): `chunk_size=25`, `overlap=5` → 54 чанка, `Hit@3 = 1.000`
- Вариант B: `chunk_size=5`, `overlap=1` → 257 чанков, `Hit@3 = 0.900`
- Выбран основной вариант A, т.к. мелкие чанки сильно теряют контекст и ухудшают качество.
- Обновление: добавлено 3 новых документа (`doc26`–`doc28`) про Python Sets.
- Переиндексация: полный перезапуск чанкинга + создание нового FAISS-индекса.
- После обновления все 3 тестовых запроса по `set` начали возвращать новые документы на первых местах (`changed = True`).

## 7. Анализ

Выбранная база знаний оказалась удачной для учебной задачи — документы короткие, но содержательные и хорошо разделены по темам.  
Чанкинг с `chunk_size=25` показал себя лучше, чем слишком мелкий (5 слов) — контекст сохраняется, а качество retrieval выше.
Эксперимент подтвердил, что избыточно мелкий чанкинг ухудшает результат.  
Обновление базы знаний работает корректно: после добавления новых документов и переиндексации retrieval сразу начал использовать свежую информацию.  
Самая показательная ошибка mini-RAG — неумение найти новые документы про `set` (пример 4). Это демонстрирует ограничение простого векторного поиска без reranking.  
Основные ограничения текущего решения: отсутствие reranking, простая конкатенация контекста вместо LLM-генерации и необходимость полной переиндексации при обновлении базы.

## 8. Итоговый вывод

В качестве базового retrieval-конвейера лучше оставить `chunk_size=25`, `overlap=5` + `all-MiniLM-L6-v2` + `IndexFlatIP`.  
Про retrieval: качество сильно зависит от размера чанка и семантической силы эмбеддингов.  
Главное про связь retrieval и mini-RAG: идеальный ответ невозможен, если retrieval вернул нерелевантный контекст. Поэтому качество retrieval — это основа всего RAG-приложения.
